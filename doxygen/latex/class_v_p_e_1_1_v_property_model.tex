\doxysubsection{Detailed Description}
This is the base model for managing all the properties and passing them to the view. 

When you create your own \char`\"{}proxy models\char`\"{}, this is the place to start\+: just subclass \mbox{\hyperlink{class_v_p_e_1_1_v_property_model}{VProperty\+Model}} and extend the new class. Have a look at existing examples of proxies.

{\bfseries{Note that in this context, the term \char`\"{}proxy model\char`\"{} does not refer to VProxy\+Model as that is another concept.}} The idea behind \char`\"{}proxy models\char`\"{} in the Qt\+Property\+Explorer framework is to provide an convenient interface which takes data as your application (or a third-\/party-\/library) provides it, and converts this data to VProperty-\/objects, manage them and produce output for the views.

In most cases, you will not need to rewrite the basic functions of QAbstract\+Item\+Model, as \mbox{\hyperlink{class_v_p_e_1_1_v_property_model}{VProperty\+Model}} provides standard implementations to work with. Thus, instead of subclassing \mbox{\hyperlink{class_v_p_e_1_1_v_property_model}{VProperty\+Model}}, it is also possible to use \mbox{\hyperlink{class_v_p_e_1_1_v_property_model}{VProperty\+Model}} directly (as it is not an abstract class). This might be more convenient in some cases.
